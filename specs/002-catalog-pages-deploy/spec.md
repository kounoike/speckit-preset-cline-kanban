# フィーチャー仕様: preset catalog の GitHub Pages 自動デプロイ

**Feature Branch**: `002-catalog-pages-deploy`
**Created**: 2026-03-29
**Status**: Draft
**Input**: User description: "preset catalog の GitHub Pages サイトを自動デプロイする"

## ユーザーシナリオ＆テスト *(必須)*

### ユーザーストーリー1 - main ブランチへのプッシュで自動デプロイ (優先度: P1)

プロジェクト管理者が preset catalog のコンテンツ（`docs/` ディレクトリ）を更新して
main ブランチにマージすると、GitHub Pages サイトが自動的に最新内容に更新される。
手動での操作は不要で、マージ後しばらくすればサイトに反映されている。

**Why this priority**: カタログの自動公開がこの機能の核心。手動デプロイのミスや忘れを防ぐ。

**Independent Test**: `docs/index.md` を更新して main にマージ後、
公開 URL にアクセスして更新内容が反映されていることで検証可能。

**Acceptance Scenarios**:

1. **Given** `docs/` ディレクトリにコンテンツの変更がコミットされている状態で、
   **When** main ブランチにプッシュ（またはPRがマージ）された、
   **Then** 5分以内に GitHub Pages サイトが新しいコンテンツを反映して公開される。

2. **Given** `docs/` 以外のファイル（例: スクリプト、テスト）のみが変更された状態で、
   **When** main ブランチにプッシュされた、
   **Then** デプロイが不要な場合はスキップされるか、または問題なく再デプロイされる。

---

### ユーザーストーリー2 - デプロイ状態の可視化 (優先度: P2)

管理者がリポジトリの CI ステータスを確認することで、
デプロイが成功したか失敗したかをすぐに把握できる。
失敗時は原因を調査して再試行できる。

**Why this priority**: デプロイ状態の可視性はトラブルシューティングに必要。P1の補完機能。

**Independent Test**: デプロイワークフローを意図的に失敗させ、
リポジトリの CI ステータスが「失敗」と表示されることで検証可能。

**Acceptance Scenarios**:

1. **Given** デプロイが正常に完了した状態で、
   **When** リポジトリのコミット履歴またはActionsタブを確認した、
   **Then** デプロイ成功のステータスマークが表示されている。

2. **Given** デプロイが何らかの原因で失敗した状態で、
   **When** リポジトリのActionsタブを確認した、
   **Then** 失敗したステップとエラー詳細が確認できる。

---

### ユーザーストーリー3 - 手動デプロイのトリガー (優先度: P3)

管理者が必要に応じてデプロイを手動で実行できる。
緊急の修正や特定のタイミングでのデプロイに対応する。

**Why this priority**: 自動トリガー以外の柔軟性を提供。緊急時のバックアップ手段。

**Independent Test**: リポジトリのActionsタブから手動でワークフローをトリガーし、
GitHub Pages サイトが更新されることで検証可能。

**Acceptance Scenarios**:

1. **Given** リポジトリの Actions ページを開いている状態で、
   **When** デプロイワークフローを手動でトリガーした、
   **Then** ワークフローが実行され GitHub Pages サイトが更新される。

---

### エッジケース

- `docs/` ディレクトリが存在しない場合、デプロイは失敗し GitHub 標準のメール通知が管理者に送られる。
- 同時に複数のプッシュが発生した場合、最新のコミットのデプロイが優先される。
- GitHub Pages のデプロイに失敗した場合（容量超過等）、エラーが CI ステータスに反映される。

## 要件 *(必須)*

### 機能要件

- **FR-001**: main ブランチへのプッシュのうち、`docs/` ディレクトリの変更を含むものに限り、自動的にデプロイワークフローが起動しなければならない（MUST）。
- **FR-002**: main ブランチの `docs/` ディレクトリを GitHub Pages のソースとして直接公開しなければならない（MUST）。ビルドステップは不要。
- **FR-003**: デプロイの成否がリポジトリの CI ステータスとして確認できなければならない（MUST）。
- **FR-004**: 管理者が手動でデプロイワークフローをトリガーできなければならない（MUST）。
- **FR-005**: デプロイが完了した後、公開 URL でサイトにアクセスできなければならない（MUST）。
- **FR-006**: デプロイ失敗時はワークフローが非ゼロの終了コードで終了し、失敗ステータスを報告しなければならない（MUST）。

### キーエンティティ

- **Preset Catalog**: `docs/` ディレクトリのコンテンツ。GitHub Pages で公開されるサイトの素材。
- **Deploy Workflow**: 自動デプロイの一連の処理。トリガー条件、実行ステップ、成否状態を持つ。
- **GitHub Pages Site**: 公開 URL、最新コンテンツ、公開状態（デプロイ中／公開中）。

## 成功基準 *(必須)*

### 測定可能な成果

- **SC-001**: main へのマージから5分以内に GitHub Pages サイトが最新コンテンツに更新される。
- **SC-002**: デプロイの成否が 100% リポジトリの CI ステータスで確認できる。
- **SC-003**: 手動デプロイトリガーから5分以内にサイトが更新される。
- **SC-004**: `docs/` ディレクトリのコンテンツが変更された場合、デプロイが自動実行される成功率が 99% 以上を維持する。

## 明確化

### Session 2026-03-29

- Q: デプロイのソース方式は → A: main ブランチの `docs/` ディレクトリを直接公開（ビルドなし）
- Q: ワークフローのトリガー範囲は → A: `docs/` ディレクトリの変更を含むプッシュ時のみ実行
- Q: デプロイ失敗時の通知方法は → A: GitHub 標準のメール通知のみ（追加インフラ不要）

## 前提条件

- リポジトリは GitHub 上にホストされており、GitHub Pages が有効化されている。
- デプロイ対象のコンテンツは `docs/` ディレクトリに配置されている。
- リポジトリの管理者は CI/CD ワークフローを設定・管理する権限を持っている。
- GitHub Actions が利用可能なリポジトリである。
- main ブランチが保護ブランチとして設定されている（PR 経由でのマージを推奨）。
