# Cline Kanban 憲法

## コア原則

### I. 仕様駆動開発

すべてのフィーチャーは、実装を開始する前に承認済みの仕様（`spec.md`）を持たなければならない（MUST）。
仕様にはユーザーストーリー、受け入れ基準、機能要件を含まなければならない（MUST）。
仕様なしの実装は禁止する。アドホックな変更は、マージ前に仕様を更新することで遡及的に記録しなければならない（MUST）。

### II. ユーザーストーリー中心

すべての実装作業は、有効な仕様内のユーザーストーリーにトレース可能でなければならない（MUST）。
各ユーザーストーリーは独立して実装・テスト可能でなければならない（MUST）。
UIの変更は、対応するユーザーストーリーを参照しなければならない（MUST）。
ユーザーストーリーに紐付かない作業はマージしてはならない（MUST NOT）。

### III. テストファースト（非交渉的）

実装の前にテストを書き、失敗することを確認しなければならない（MUST）（Red-Green-Refactorサイクル）。
これはすべての新機能とバグ修正に適用される。
例外はPRの説明に明示的な書面による正当化が必要である。
全テスト通過がすべてのマージの前提条件である（MUST）。

### IV. 段階的デリバリー

フィーチャーはユーザーストーリーの優先度順（P1 → P2 → P3）に独立した機能インクリメントとして提供しなければならない（MUST）。
各インクリメントは次を始める前に単独でデプロイ・デモ可能でなければならない（MUST）。
MVPはP1ストーリーのみで到達可能でなければならない（MUST）。

### V. シンプルさ（YAGNI）

投機的な機能や抽象化は禁止する（MUST NOT）。
複雑さは現在の具体的な要件によって正当化されなければならない（MUST）。
受け入れ基準を満たす最もシンプルな解を優先する。
シンプルさからの逸脱はすべて、該当する`plan.md`の「複雑さ追跡」テーブルに記録しなければならない（MUST）。

## 開発ワークフロー

- すべてのフィーチャー作業はSpecKitワークフローに従わなければならない（MUST）:
  `/speckit.specify` → `/speckit.plan` → `/speckit.tasks` → `/speckit.implement`
- フィーチャーブランチは連番を使用した `###-短い説明` の形式で命名しなければならない（MUST）。
- `tasks.md`の各タスクは、説明的なコミットメッセージとともに個別またはロジカルグループとしてコミットしなければならない（MUST）。
- プルリクエストはレビュー依頼前に、仕様を参照し`plan.md`の憲法チェックをパスしなければならない（MUST）。
- `main`へのマージには、少なくとも1名のレビュアーの承認と全テスト通過が必要である（MUST）。

## ガバナンス

この憲法はすべての他のプラクティスや非公式の合意に優先する。
Cline Kanbanプロジェクトにおける開発ルールの権威ある情報源である。

**改正手順**:
1. `.specify/memory/constitution.md`を変更しバージョンを更新するPRを開く。
2. PRの説明に根拠と既存のスペック・プランへの影響を記載する。
3. マージ前に憲法管理者（プロジェクトオーナー）の承認を得る。
4. マージ後、同期影響レポートに従って影響を受けるテンプレートの変更を伝播させる。

**バージョニングポリシー**（セマンティックバージョニング）:
- MAJOR: 後方互換性のない原則の削除または再定義。
- MINOR: 新しい原則またはセクションの追加、または実質的なガイダンスの拡張。
- PATCH: 明確化、文言修正、または非意味論的な改良。

**コンプライアンスレビュー**: すべてのPRの説明に「憲法チェック」セクションを含め、
コンプライアンスの確認または承認済み例外を文書化しなければならない（MUST）。
